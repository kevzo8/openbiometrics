Adding face recognition to your app shouldn't require a PhD in cloud infrastructure. But if you've tried AWS Rekognition or Azure Face API, you know the onboarding pain: IAM policies, billing alerts, SDK incompatibilities, and the nagging question — *where is my users' biometric data going?*

This guide compares the major face recognition APIs head-to-head and explains when each makes sense.

## The Landscape

There are three categories of face recognition solutions:

1. **Cloud APIs** (AWS, Azure, Google) — pay per call, fully managed, no infrastructure
2. **SaaS APIs** (Luxand, Face++, Kairos) — simpler than cloud giants, subscription pricing
3. **Open-source** (OpenBiometrics, InsightFace, DeepFace) — self-hosted, free, you own everything

Most developers default to cloud APIs because they seem easiest. But are they?

## Time to First API Call

This is what actually matters. You have a feature to ship. How fast can you go from zero to working face detection?

### AWS Rekognition: ~2 hours

```
1. Create AWS account (credit card, phone verification)
2. Navigate to IAM → Create user → Attach policy
3. Create access key + secret
4. pip install boto3
5. Configure AWS CLI or environment variables
6. Learn Rekognition API (it's different from every other AWS service)
7. Write code, parse their response format
8. Handle region selection, error codes, rate limits
```

### Azure Face API: ~1 hour

```
1. Create Azure account
2. Create resource group
3. Create Cognitive Services resource
4. Copy endpoint URL + API key
5. pip install azure-cognitiveservices-vision-face
   (deprecated — now use azure-ai-vision-face)
   (wait, that's also deprecated — check docs again)
6. Handle their SDK breaking changes
7. Write code
```

### OpenBiometrics: ~30 seconds

```bash
pip install openbiometrics
```

```python
from openbiometrics_sdk import OpenBiometrics

ob = OpenBiometrics(api_key="ob_live_xxx")
result = ob.faces.detect("photo.jpg")
print(f"Found {len(result['faces'])} faces")
```

Or self-hosted (also ~30 seconds):

```bash
git clone https://github.com/openbm/openbiometrics && cd openbiometrics
cd engine && pip install -e . && cd .. && python download_models.py --module face
cd api && uvicorn app.main:app --port 8000
```

## Pricing at Scale

Cloud APIs charge per image. It adds up fast.

| Monthly Volume | AWS Rekognition | Azure Face | Luxand | OpenBiometrics (self-hosted) | OpenBiometrics (hosted) |
|---|---|---|---|---|---|
| 1,000 | Free (trial) | Free | $9 | **$0** | Free tier |
| 10,000 | $10 | ~$10 | $39 | **$0** | ~$19 |
| 100,000 | $100 | ~$100 | $99 | **$0** | ~$49 |
| 1,000,000 | $1,000 | ~$1,000 | $1,949 | **$0** | ~$199 |
| 10,000,000 | $6,000 | ~$6,000 | Custom | **$0** | Custom |

**At 1M calls/month, you're paying $12,000+/year to send your users' faces to Amazon.** With OpenBiometrics self-hosted, the cost is whatever your server costs — typically $50-200/month for a GPU instance that handles millions of calls.

## Data Sovereignty

This is the elephant in the room. When you use AWS Rekognition:

- Your users' face images travel to AWS servers
- They're processed in a specific AWS region
- You're trusting Amazon with biometric data
- GDPR compliance becomes *your* problem to prove
- Some jurisdictions (healthcare, government, banking) prohibit this entirely

With OpenBiometrics self-hosted:

- **Data never leaves your infrastructure**
- You choose where it runs — your datacenter, your cloud, your Raspberry Pi
- GDPR compliance is straightforward — you control everything
- Air-gapped deployments for sensitive environments
- No third-party data processing agreements needed

## Accuracy

Let's be honest: cloud providers have better models. They train on billions of images with massive compute budgets.

| Provider | Face Detection | Face Recognition | Age Estimation |
|---|---|---|---|
| AWS Rekognition | Excellent | Excellent | Good |
| Azure Face | Excellent | Excellent | Good |
| OpenBiometrics (community) | Good (YuNet, 0.88 AP) | Good (SFace, 99.4% LFW) | Good (ViT, 4.5yr MAE) |
| OpenBiometrics (premium) | Excellent (partner models) | Excellent (partner models) | Excellent |

For most use cases — age verification, 2FA, visitor management, basic KYC — the accuracy difference doesn't matter. You don't need 99.99% when 99.4% does the job.

The premium tier bridges the gap for high-security applications by partnering with NIST-ranked model providers.

## When to Use What

### Use AWS/Azure when:
- You're already deep in their ecosystem
- You need the absolute highest accuracy
- You process < 10,000 images/month (free tier)
- Data sovereignty isn't a concern
- You have a cloud team managing infrastructure

### Use Luxand/SaaS when:
- You want simple subscription pricing
- You need a quick prototype
- You don't want to manage any infrastructure

### Use OpenBiometrics when:
- Data sovereignty matters (GDPR, healthcare, government)
- You need to run on-prem or at the edge
- Cost at scale matters (>100K calls/month)
- You want to see and control what models run
- You need the full stack (face + documents + liveness + video) in one API
- You're building a product, not a prototype

## Getting Started

```bash
pip install openbiometrics-engine
python -c "
from openbiometrics import BiometricKernel, BiometricConfig, FaceConfig
kernel = BiometricKernel(BiometricConfig(face=FaceConfig(ctx_id=-1)))
print('Models:', kernel.face.loaded_models)
"
```

Or try the hosted API:
```bash
curl -X POST https://api.openbiometrics.dev/api/v1/detect \
  -H "Authorization: Bearer ob_live_demo" \
  -F "image=@selfie.jpg"
```

Check which models are running and their licenses:
```bash
curl https://api.openbiometrics.dev/api/v1/admin/health
```

Every response includes the `X-OpenBiometrics-Version` header so you always know what version you're running.
